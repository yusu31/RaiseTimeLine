# 機能詳細設計: タイムラインの新着チェック（F-12）

## 概要

タイムライン画面が開いている間、一定間隔で新着投稿の有無を確認し、あれば通知バナーを表示する機能（F-12）を実装する。

複数ユーザーが同時に使う場合、投稿・編集・削除のたびに全員へリアルタイムに同期する仕組み（WebSocketなど）は、
通知やタイムライン表示が絶えず変化し、かえってユーザビリティを損なう懸念がある。X（旧Twitter）のように
「新着があることだけ控えめに知らせ、実際に見るかどうかはユーザーの任意」という**定期チェック＋通知バナー方式**を採用した。

## 技術方針

### 軽量チェック用エンドポイントと新着取得エンドポイントを分離した理由

`GET /api/posts/new-count`は`posts`テーブル単体への`COUNT(*) WHERE id > #{afterId}`のみで完結し、JOIN不要のため
既存の「単純なCRUD/単票SELECTはアノテーション方式」という慣習にそのまま乗る。フロントエンドが30秒ごとに
ポーリングしてもDBへの負荷が小さい。

一方、実際に新着投稿の中身（投稿者・本文・画像URLなど）を取得する`GET /api/posts/new`は`posts`×`users`のJOINを伴うため、
バナークリック時（頻度が低い）にのみ呼び出す。既存の`GET /api/posts`（offsetページネーション前提の`PostListResponse`）を
拡張するのではなく、`page`を持たない専用の`NewPostsResponse`を新設した。理由は、`afterId`モードと`page`/`size`モードという
2つの矛盾したページネーション概念を1つのレスポンス型に混在させると、読み手にとって`page`の意味が曖昧になるため。

### 新着判定のマーカーに投稿id（連番PK）を使う理由

既存の並び順（`ORDER BY created_at DESC, id DESC`）が既にidをtie-breakerとして使っている。`created_at`ではなくidを
マーカーにすることで、タイムスタンプの精度・衝突の問題を避けられ、かつ編集（`updated_at`更新）を新着として
誤検知しない（X/Twitterも編集で新着扱いにはしない）。

マーカーの実体はフロントエンドが保持する投稿配列中の最大id（`Math.max(...posts.map(p => p.id))`）。配列全体から
算出することで、「もっと見る」で末尾に古い投稿を追加しても値が変わらず安全。

### ポーリング間隔（30秒）の根拠

- 短すぎる（5〜10秒）とWebSocketに近い頻度になり、「絶えず変化する画面を避けたい」という設計意図に反する
- 長すぎる（60秒以上）と新着に気づくまでの遅延が目立つ
- 100人規模でも軽量なCOUNT(*)クエリを30秒間隔で叩く程度なら、単一テーブルの主キー範囲検索で十分に捌ける負荷

### タブが非アクティブな間はポーリングを停止する

`document.visibilitychange`を利用し、タブが非表示の間は`clearInterval`でポーリングを止め、再表示時に即座に
1回チェックしてからポーリングを再開する。実装コストが小さい一方、開きっぱなしの非アクティブタブが多いほど
サーバー負荷の抑制効果が大きいため、今回のスコープに含めた。

## エンドポイント仕様

API一覧・レスポンス例は [docs/design.md](../design.md) を参照。ここでは実装レベルの詳細のみ記載する。

| メソッド | パス | 処理概要 |
|---|---|---|
| GET | /api/posts/new-count | `posts`単体へのCOUNT(*)のみ。JOIN不要 |
| GET | /api/posts/new | `size+1`件取得して`hasMore`を判定する軽量方式（`getTimeline`と同じ流儀）。取得上限は50件 |

`hasMore`は取得件数が上限（50件）を超えた場合に`true`になる。上限を超える新着が溜まった状態でバナーをクリックした場合、
最新50件のみが反映される（既知の制限、後述）。

## 設計判断（要確認事項）

**offsetページネーション（「もっと見る」）とバナー経由の新着追加が理論上ウィンドウずれを起こす既知の制限**

DBに50件（P1〜P50）あり、クライアントは`page=0`で最新20件（P50〜P31）を表示中に5件（P51〜P55）が追加された場合、
バナーをクリックせずに「もっと見る」（`page=1`）を押すと、サーバーはその時点のDB全体（55件）の中でoffset=20〜39番目を
返すため、既に表示済みの範囲と一部重複する投稿が返ってくることがある。

これはoffsetページネーション自体が持つ「挿入によるウィンドウずれ」という既知の欠陥であり、**このポーリング機能を
追加する前から`GET /api/posts`のoffset方式に内在していた問題**（今回新たに作り込むものではない）。今回のスコープでは
修正せず、既知の制限として記録するに留める。根本修正（`GET /api/posts`自体をidベースのカーソル方式に置き換える）は
将来の拡張候補とする。

## フロントエンド実装

`frontend/src/pages/TimelinePage.tsx`に既存の状態管理パターンのまま追加した（新規カスタムフックは作らない。
投稿関連の状態が全てこのコンポーネントに集約されている既存方針を維持するため）。

### ファイル構成

- `types/post.ts`: `NewPostsCountResponse`/`NewPostsResponse`型を追加
- `api/postApi.ts`: `fetchNewPostsCount`/`fetchNewPosts`を追加。既存の`fetchTimeline`と同じ「フックを引数に取る関数」形式
- `components/NewPostsBanner.tsx`: バナー表示用の提示コンポーネント（`count`/`isLoading`/`onClick`を受け取る）
- `pages/TimelinePage.tsx`: `latestKnownId`（`useMemo`で配列全体から算出）、ポーリング用`useEffect`、
  `handleShowNewPosts`（バナークリック時の反映処理）を追加

### エラーハンドリングの非対称性

バックグラウンドの定期チェック（`fetchNewPostsCount`）が失敗しても画面の`error`ステートには反映しない
（黙って次回のポーリングに委ねる）。一方、バナークリック起点の`fetchNewPosts`が失敗した場合は`handleLoadMore`と
同様に`setError`で明示的にユーザーへ表示する。これは「背景処理のエラーで読書体験を妨げない」という意図的な設計判断。

## 既知の制限

- offsetページネーションとの競合（前述の「設計判断」参照）
- 新着投稿が50件を超えて溜まった状態でバナーをクリックすると、最新50件のみが反映される
- タブを閉じたまま長時間経過した場合の新着件数は、再度開いた時点でのポーリングで確認される（プッシュ通知ではない）
