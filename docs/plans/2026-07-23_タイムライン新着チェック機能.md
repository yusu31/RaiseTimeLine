# タイムラインの新着チェック機能（F-12）実装計画

> Claude Codeのplanモードで承認された実装計画のアーカイブ。元ファイル: `C:\Users\3fort\.claude\plans\ai-websocket-100-x-twitter-ticklish-toucan.md`(実装・マージ済み、Issue #19 → PR #20)

## Context（なぜこの変更を行うか）

指導者から「WebSocketで常に同期しようとしていると思うが、100人規模のユーザーが投稿・編集・削除するたびに全員へリアルタイム同期すると、通知やタイムライン表示が絶えず増えてユーザビリティを損なう。X/Twitterのように、任意のタイミングで更新した時だけ最新が取れる、もしくは一定時間経った時に最新が取れる形でよい」という設計方針の指示があった。

**前提の裏取り結果**: 実際にはWebSocketはバックエンド・フロントエンドどちらにも実装されておらず、`docs/tech-stack.md`にも「WebSocket | リアルタイム更新（通知など）はMVP後に検討」と明記された、そもそも採用していない技術だった。現状のタイムラインは`TimelinePage.tsx`がマウント時に1回`GET /api/posts`を叩くだけで、自動更新の仕組みは何もない。つまりこれは既存実装の修正ではなく、**まだ存在しない「タイムライン自動更新」機能をゼロから設計・実装する**という位置づけになる。

ユーザーとの相談の結果、以下の方針で確定した:
- 方式は**定期チェック＋通知バナー方式（X/Twitter方式）**を採用する
- F-03（コメント）・F-04（いいね）より**先に**、今回実装する
- 新機能として`docs/requirements.md`にF-12を追加する
- （申し送り）Phase順の食い違いが今後生じた場合、都度確認を取らず講師指示を優先してよいとの指示あり。ただし技術的前提の誤りは引き続き実装前に裏取りする

## 採用するアーキテクチャ

- **軽量チェック用エンドポイント**: `GET /api/posts/new-count?afterId={id}` — `posts`単体への`COUNT(*) WHERE id > #{afterId}`（JOIN不要）。既存の「単純なCRUD/単票SELECTはアノテーション方式、JOINを伴うものだけXMLマッパー」という使い分けにそのまま乗る
- **新着取得エンドポイント**: `GET /api/posts/new?afterId={id}` を**別エンドポイントとして新設**（既存`GET /api/posts`の拡張はしない）。理由: `PostListResponse`は`page`/`hasNext`というoffsetページネーション前提のフィールドを持ち、ここに`afterId`モードを混ぜると2つの矛盾したページネーション概念が同居してしまう。新設する`NewPostsResponse`は`posts`/`hasMore`のみを持つ
- **新着判定のマーカーには投稿id（連番PK）を使う**（`created_at`は使わない）。既存の並び順が`ORDER BY created_at DESC, id DESC`でidを既にtie-breakerとして使っており、タイムスタンプの衝突・精度問題を避けられる。また編集（`updated_at`更新）を新着として誤検知しない
- **ポーリング間隔は30秒**。短すぎるとWebSocketに近い頻度になり講師の設計意図に反し、長すぎると鮮度感が失われるため
- **タブ非アクティブ時はポーリングを停止し、再表示時に即時チェック**（`document.visibilitychange`対応。実装コストが小さく、サーバー負荷抑制という講師の意図に直接効くため今回のスコープに含める）
- **offsetページネーション（「もっと見る」）とバナー経由の新着追加が理論上ウィンドウずれを起こす既知の制限は、今回のスコープでは許容し、ドキュメントに明記するのみに留める**（この問題はポーリング機能追加前から`GET /api/posts`のoffset方式に内在するものであり、今回新たに作り込むものではない）

## 実装ファイル

### バックエンド

| ファイル | 変更内容 |
|---|---|
| `backend/src/main/java/com/raisetech/raisetimeline/mapper/PostMapper.java` | `@Select("SELECT COUNT(*) FROM posts WHERE id > #{afterId}") long countNewerThan(@Param("afterId") long afterId);` を追加（アノテーション方式）。加えて `List<PostDetail> selectNewerThan(@Param("afterId") long afterId, @Param("limit") int limit);` のシグネチャのみ追加（実装はXML側） |
| `backend/src/main/resources/mapper/PostMapper.xml` | `selectNewerThan`を追加。`selectTimeline`と同じSELECT列・JOINを流用し、`WHERE p.id > #{afterId} ORDER BY p.id DESC LIMIT #{limit}`とする（`created_at`ではなく`id`のみで比較・整列） |
| `backend/src/main/java/com/raisetech/raisetimeline/response/NewPostsCountResponse.java`（新規） | `public record NewPostsCountResponse(long count) {}` |
| `backend/src/main/java/com/raisetech/raisetimeline/response/NewPostsResponse.java`（新規） | `public record NewPostsResponse(List<PostResponse> posts, boolean hasMore) {}` |
| `backend/src/main/java/com/raisetech/raisetimeline/service/PostService.java` | `NEW_POSTS_FETCH_LIMIT = 50`定数を追加。`getNewPostsCount(long afterId)`＝`countNewerThan`のラップ。`getNewPosts(long afterId)`＝`selectNewerThan(afterId, limit+1)`で既存の`size+1`パターンにより`hasMore`判定し、既存の`toResponse(PostDetail)`を再利用。`afterId`は`Math.max(afterId, 0)`で正規化 |
| `backend/src/main/java/com/raisetech/raisetimeline/controller/PostController.java` | `@GetMapping("/new-count")`と`@GetMapping("/new")`を追加。どちらも`@RequestParam(defaultValue = "0") long afterId`を受け取る（既存の`page`/`size`と同じ primitive + defaultValue パターン） |
| `backend/src/test/java/com/raisetech/raisetimeline/controller/PostControllerIntegrationTest.java` | 既存ファイルに追加（新規ファイルは作らない）。ケース: ①新着0件でcount=0 ②新着N件でcount=N ③`/new`がafterIdより新しい投稿のみid降順で返す ④afterIdが最新id以上で`posts`空・`hasMore=false` ⑤上限超過で`hasMore=true` ⑥`/new-count`・`/new`が`/{id}`の数値変換エラーとして誤処理されないこと（ルーティング上の唯一の落とし穴） ⑦未認証401 |

`SecurityConfig`・`GlobalExceptionHandler`は変更不要（新エンドポイントは`/api/posts/**`の既存認証必須ルールに自動的に含まれ、新しい例外パターンも発生しない）。

**実装時に見落としやすい点**:
- バックグラウンドの定期チェック失敗は画面の`error`ステートに反映しない（黙って次回ポーリングに委ねる）。バナークリック起点の取得失敗は`handleLoadMore`と同様に`setError`で明示表示する（背景処理のエラーで読書体験を妨げない、という意図的な非対称）
- `latestKnownId`が変化するたび（バナークリック後・自分の投稿作成後）に`useEffect`が再実行され、新しい`cancelled`フラグで即時再チェックが走る
