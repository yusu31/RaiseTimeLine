## 1. システム構成

```
+-------------------+        +----------------------+        +----------------+
|   ブラウザ         |  REST  |   バックエンド        |  JPA   |  データベース   |
|   React + Vite    | <----> |   Spring Boot        | <----> |  PostgreSQL    |
|   (ポート 5173)    |  JSON  |   (ポート 8080)       |        |  (ポート 5432)  |
+-------------------+        +----------------------+        +----------------+
        |                            |
        |  Authorization ヘッダーに    |  Spring Security が
        |  アクセストークンを添付       |  トークンを検証
        v                            v
   ログイン後の全リクエストで JWT を送る
```

- フロントエンドは REST API（JSON）でバックエンドと通信する
- 画像は `multipart/form-data` でアップロードし、**AWS S3 に保存する（確定）**。アイコン画像（Phase 2）も S3 に保存する
  - ローカル開発時の扱い（ローカル保存での代替 or 開発用バケット）は環境構築時に決定する
- DBマイグレーション（テーブル作成・変更）は Flyway が起動時に自動実行する

## 2. 認証方式（JWT）

| トークン | 有効期限（目安） | 保管場所 | 役割 |
|---------|---------------|---------|------|
| アクセストークン | 30分 | フロントエンド（保管場所は実装時に比較検討） | 毎回のAPIリクエストの `Authorization: Bearer <token>` ヘッダーに添付する「入館証」 |
| リフレッシュトークン | 14日 | フロントエンド + DB（refresh_tokens テーブル） | アクセストークン失効時に再発行を受けるための「再発行券」。ログアウト時にDBから削除して無効化する |

### 認証の流れ

1. `POST /api/auth/login` にメールアドレス・パスワードを送る
2. サーバーはパスワードを検証（BCryptで照合）し、アクセストークンとリフレッシュトークンを返す
3. 以降のAPIリクエストにはアクセストークンを添付する
4. アクセストークンが期限切れになったら `POST /api/auth/refresh` で再発行する
5. ログアウト時は `POST /api/auth/logout` でリフレッシュトークンを無効化する

## 3. APIエンドポイント一覧

### 認証（Phase 1）

| メソッド | パス | 説明 | 認証 |
|---------|------|------|------|
| POST | /api/auth/login | ログイン（トークン発行） | 不要 |
| POST | /api/auth/refresh | アクセストークン再発行 | 不要（リフレッシュトークン必須） |
| POST | /api/auth/logout | ログアウト（リフレッシュトークン無効化） | 必要 |
| POST | /api/auth/signup | ユーザー登録 | 不要 |
| GET | /api/hello | 認証確認用の暫定エンドポイント（`{"message": "Hello, ..."}`を返す） | 必要 |

> **signupについて:** 当初はPhase 2（ユーザー登録画面と合わせて実装）の予定だったが、認証機能をひとまとまりで完成させたいと考え、認証フェーズで先行実装した。
> **`GET /api/hello`について:** フロントエンドを今回実装しない都合上、「ログイン後の画面」の代わりに用意した暫定API。認証が通ればトークンの有無に応じて200/401が返ることを確認できる。フロントエンド実装時（タイムライン画面ができた時点）に削除予定。詳細は [docs/features/auth.md](./features/auth.md) を参照。

### 投稿（Phase 1）

| メソッド | パス | 説明 | 認証 |
|---------|------|------|------|
| GET | /api/posts | タイムライン取得（新着順・ページネーション） | 必要 |
| GET | /api/posts/{id} | 投稿詳細取得 | 必要 |
| POST | /api/posts | 投稿作成（本文＋画像1枚まで、multipart/form-data） | 必要 |
| PUT | /api/posts/{id} | 投稿編集（自分の投稿のみ） | 必要（**Phase 2**） |
| DELETE | /api/posts/{id} | 投稿削除（自分の投稿のみ） | 必要（**Phase 2**） |

**GET /api/posts のクエリパラメータ**

| パラメータ | 型 | デフォルト | 説明 |
|-----------|----|-----------|------|
| page | 数値 | 0 | ページ番号（0始まり） |
| size | 数値 | 20 | 1ページあたりの件数 |

### コメント（Phase 1）

| メソッド | パス | 説明 | 認証 |
|---------|------|------|------|
| GET | /api/posts/{postId}/comments | コメント一覧取得 | 必要 |
| POST | /api/posts/{postId}/comments | コメント投稿 | 必要 |

### いいね（Phase 1）

| メソッド | パス | 説明 | 認証 |
|---------|------|------|------|
| POST | /api/posts/{postId}/like | いいねする | 必要 |
| DELETE | /api/posts/{postId}/like | いいねを取り消す | 必要 |

### 画像（Phase 1）

画像は AWS S3 に保存する（確定）。APIレスポンスの `imageUrl` には S3 のオブジェクトURLを設定する想定。
ローカル開発時の配信方法は環境構築時に決定する。

### ユーザー・プロフィール（Phase 2）

| メソッド | パス | 説明 | 認証 |
|---------|------|------|------|
| GET | /api/users?q={keyword} | ユーザー検索（ユーザー名・表示名の部分一致） | 必要 |
| GET | /api/users/{id} | プロフィール取得（そのユーザーの投稿一覧含む） | 必要 |
| PUT | /api/users/me | 自分のプロフィール編集（表示名・自己紹介・アイコン） | 必要 |

### フォロー（Phase 3）

| メソッド | パス | 説明 | 認証 |
|---------|------|------|------|
| POST | /api/users/{id}/follow | フォローする | 必要 |
| DELETE | /api/users/{id}/follow | フォロー解除 | 必要 |
| GET | /api/posts?timeline=following | フォロー中ユーザーのみのタイムライン取得 | 必要 |

## 4. レスポンス例

### GET /api/posts（タイムライン取得）

```json
{
  "posts": [
    {
      "id": 1,
      "content": "今日はSpring Securityの講義。JWTむずかしい…",
      "imageUrl": null,
      "author": {
        "id": 2,
        "displayName": "鈴木"
      },
      "likeCount": 12,
      "commentCount": 3,
      "likedByMe": true,
      "createdAt": "2026-07-17T10:30:00"
    }
  ],
  "page": 0,
  "hasNext": true
}
```

> **ポイント:** いいね数・コメント数・「自分がいいね済みか（likedByMe）」をタイムライン取得時にまとめて返す。
> フロントエンドが投稿ごとに追加リクエストを送らなくて済み、N+1問題（リクエストの多発）を防ぐ。

### POST /api/auth/login（ログイン）

```json
{
  "accessToken": "eyJhbGciOiJIUzI1NiJ9...",
  "refreshToken": "8f14e45f-ceea-467f-a34e-...",
  "user": {
    "id": 2,
    "displayName": "鈴木",
    "email": "suzuki@example.com"
  }
}
```

## 5. エラーレスポンス形式

すべてのエラーは共通の形式で返す（フロントエンドの処理を統一するため）。

```json
{
  "status": 400,
  "error": "Bad Request",
  "message": "本文は1〜280文字で入力してください"
}
```

### HTTPステータスコード一覧

| コード | 意味 | 使いどころ |
|-------|------|-----------|
| 200 | OK | 取得・更新の成功 |
| 201 | Created | 投稿・コメントなどの作成成功 |
| 400 | Bad Request | バリデーションエラー（文字数超過・画像形式違反など） |
| 401 | Unauthorized | 未ログイン・トークン期限切れ |
| 403 | Forbidden | 権限なし（他人の投稿の編集・削除など） |
| 404 | Not Found | 存在しない投稿・ユーザーへのアクセス |
| 409 | Conflict | 競合（同じ投稿への二重いいね、登録済みメールアドレスなど） |
| 500 | Internal Server Error | サーバー内部エラー |

## 6. セキュリティ設計

| 項目 | 方針 |
|------|------|
| パスワード | BCrypt でハッシュ化して保存（平文・可逆な暗号化は禁止） |
| SQLインジェクション | Spring Data JPA のプリペアドステートメントを使用 |
| XSS | React の自動エスケープに任せる。`dangerouslySetInnerHTML` は使用しない |
| CSRF | JWTを `Authorization` ヘッダーで送るため、Cookie の自動送信に依存しない構成とする |
| CORS | `http://localhost:5173`（本番はフロントのドメイン）のみ許可 |
| ファイルアップロード | 拡張子とContent-Typeを検証し、JPEG/PNG・5MB以内のみ受け付ける。保存ファイル名はサーバー側で採番（UUID）する |
