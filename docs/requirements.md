## 1. プロジェクト概要

### 目的

X（旧Twitter）のタイムライン形式を参考にした、テキストベースのSNS風Webアプリを作成する。
複数のユーザーが投稿・コメント・いいねで交流できる最小限のSNSを、
要件定義 → 設計 → 実装 → デプロイの一連の流れで開発し、次のスキルを習得する。

- React + Spring Boot + PostgreSQL による3層構成のWebアプリケーション開発
- ユーザー認証（JWT）・リレーショナルデータベース設計・REST API設計の実践
- AWS環境へのデプロイ経験

### 背景

- プログラミングスクール RaiseTech の課題として、講師の講義進行に合わせて開発する
- 前回課題（Trello風タスク管理アプリ）と同じ技術スタック（Spring Boot + React + PostgreSQL）で、認証・複数ユーザー・画像アップロードなど新しい要素に挑戦する
- 利用者は受講生ないし個人を想定するが、**複数のユーザーが使うことを前提に設計する**（第三者がいいね・コメントする状況があり得る）

### Xとの差別化ポイント（意図的に作らないもの）

| 項目 | 方針 |
|------|------|
| インプレッション数（表示回数） | **表示しない** |
| リツイート（リポスト）機能 | **作らない** |

## 2. 前提条件・制約事項

### 前提条件

- 開発環境は Windows 11 PC + Cursor / Claude Code エディタとする
- バックエンドは Java + Spring Boot（REST API）、データベースは PostgreSQL を使用する
  - ローカル開発: Docker（`docker-compose up -d`）で PostgreSQL を起動
  - 認証: Spring Security + JWT（アクセストークン + リフレッシュトークン）
  - DBマイグレーション: Flyway で管理する
- フロントエンドは React + TypeScript（Vite）で作成する
- 画像（投稿画像・アイコン画像）の保存先は本番環境では **AWS S3** とする（確定）。ローカル開発中はファイルシステム保存とし、`StorageService`抽象化によりS3実装へ差し替え可能な設計にする（デプロイフェーズで切替）
- 使用ライブラリのバージョンは前回課題 TaskManagement と同じものを使う（詳細は tech-stack.md）
- ポート番号は固定（フロントエンド 5173 / バックエンド 8080 / DB 5432。TaskManagementと同一）とし、変更しない

### 制約事項

- インプレッション数は表示しない・リツイート機能は作らない（差別化ポイントのため）
- 個人学習用途のため、大規模トラフィックは想定しない
- 画像は JPEG / PNG 形式のみ対応。動画投稿には対応しない
- 機能追加は Phase 順を厳守する（Phase 1 が完成するまで Phase 2 に着手しない）
- 有料APIを呼び出す実装を行う前に、必ずユーザーに確認を取る（課金が発生するため）

## 3. 対象ユーザー（ペルソナ）

### ペルソナ 1：RaiseTech受講生・鈴木さん（26歳）

| 項目 | 内容 |
|------|------|
| 職業 | 転職を目指してスクールで学習中（前職は販売職） |
| 利用目的 | 学習の進捗や気づきを投稿して、他の受講生と励まし合いたい |
| ITリテラシー | X・Instagramは日常的に使う。開発は学習中 |
| 利用環境 | PCブラウザ |
| 価値観 | 気軽に短文で発信したい。反応（いいね・コメント）がモチベーションになる |

### ペルソナ 2：個人ユーザー・高橋さん（32歳）

| 項目 | 内容 |
|------|------|
| 職業 | 会社員。趣味の記録用にSNSを使いたい |
| 利用目的 | 日々の記録や写真を、身内だけの落ち着いた場所に投稿したい |
| ITリテラシー | SNS利用歴は長いが、大手SNSの「数字（表示回数）に追われる感じ」が苦手 |
| 利用環境 | PCブラウザ |
| 価値観 | インプレッション数や拡散機能のない、静かなタイムラインが心地よい |

### ユーザーが抱える課題と解決

- 大手SNSはインプレッション数や拡散が前提で、気軽な発信がしづらい
  → 表示回数や拡散機能をあえて持たない、投稿・コメント・いいねだけのシンプルなタイムラインを提供する
- 学習仲間・身内だけの小さなコミュニティで使えるシンプルな投稿場所がない
  → 複数ユーザーで使える最小限のSNSとして提供する（コメント数・いいね数は表示するので反応は受け取れる）

## 4. 機能要件

主要機能の概要を以下に示す。**受け入れ条件・ユースケースを含む詳細は [機能一覧（features.md）](./features.md) を参照。**

| No | 機能名 | 概要 | Phase |
|----|--------|------|-------|
| F-01 | ログイン機能 | メールアドレス + パスワードでの認証（JWT）。ログイン・ログアウト | 1 |
| F-02 | タイムライン機能 | 280文字制限のテキスト投稿、全ユーザーの投稿を新着順で一覧表示 | 1 |
| F-03 | コメント機能 | 投稿へのコメント、コメント数表示 | 1 |
| F-04 | いいね機能 | 投稿へのいいね／取消（トグル）、いいね数表示 | 1 |
| F-05 | 画像投稿機能 | 投稿への画像添付（1枚まで・JPEG/PNG・5MB以内） | 1 |
| F-06 | ユーザー登録 | メールアドレス・パスワードによる新規アカウント作成 | 2 |
| F-07 | プロフィール画面 | 表示名・自己紹介・自分の投稿一覧の表示 | 2 |
| F-08 | 投稿の削除・編集 | 自分の投稿に限り削除・編集できる | 1（※） |
| F-09 | キーワード検索 | 投稿本文のキーワード検索 | 2 |
| F-10 | ユーザー検索 | ユーザー名・表示名でユーザーを検索し、プロフィールへ遷移できる | 2 |
| F-11 | フォロー機能 | フォロー/フォロワー関係の管理、タイムラインの「全体/フォロー中」タブ切替 | 3 |

> **（※）F-08について:** 当初はPhase 2の予定だったが、投稿機能（作成・表示）の基本操作の一部として捉え、F-02・F-05と合わせてPhase 1で実装することにした。

## 5. 非機能要件

### 5.1 対応ブラウザ

| ブラウザ | バージョン |
|---------|-----------|
| Google Chrome | 最新版 |
| Microsoft Edge | 最新版 |

### 5.2 性能

| 項目 | 目標 |
|------|------|
| 画面初期表示 | 3秒以内 |
| 各操作のレスポンス | 1秒以内 |
| タイムライン表示 | ページネーションにより段階的に読み込む（一度に全件取得しない） |

### 5.3 レスポンシブ対応

- PC（1024px以上）を対象とする
- モバイル最適化は将来の拡張候補とする

### 5.4 セキュリティ

| 項目 | 対策 |
|------|------|
| パスワード保管 | 平文では保存せず、BCryptでハッシュ化して保存する |
| 認証 | Spring Security + JWT（アクセストークン + リフレッシュトークン） |
| SQLインジェクション | Spring Data JPA のプリペアドステートメントを使用（SQL文字列の手組みをしない） |
| XSS | React の自動エスケープを利用。`dangerouslySetInnerHTML` は使用しない |
| CORS | フロントエンド（localhost:5173）からのアクセスのみ許可する設定を行う |

### 5.5 データ・通信

| 項目 | 内容 |
|------|------|
| データ永続化 | PostgreSQL（マイグレーションは Flyway で管理） |
| 画像保存 | 本番はAWS S3（確定）。ローカル開発中はファイルシステム保存で代替し、`StorageService`抽象化でS3に切替可能な設計にする |
| 通信 | REST API（JSON）。ローカル開発は http、本番は https を想定 |
| ホスティング | AWS。EC2 + RDS + ALB を前提とした構成図は [infrastructure.md](./infrastructure.md) を参照（サーバー構築方式の詳細はデプロイフェーズで確定） |

## 10. リスクと対策

| # | リスク | 影響 | 対策 |
|---|--------|------|------|
| 1 | JWTの盗難（XSS攻撃） | 他人になりすまされる | Reactの標準機能（自動エスケープ）でXSSを防ぐ。トークンの保管場所は実装フェーズで比較検討する |
| 2 | 二重いいね（連打・不具合） | いいね数が不正確になる | DBの likes テーブルに（user_id, post_id）のユニーク制約を設定し、仕組みで防ぐ |
| 3 | 巨大な画像のアップロード | サーバーのディスク圧迫・表示速度低下 | 5MB・JPEG/PNG のみに制限し、超過時はエラーを返す |
| 4 | いいね数・コメント数取得でのN+1問題 | タイムライン表示が遅くなる | 集計クエリ（COUNT + GROUP BY）でまとめて取得する設計にする |
| 5 | 機能の作りすぎ（スコープ膨張） | Phase 1 が完成しない | Phase 順を厳守。Phase 1 完成（動作確認済み）まで追加機能に着手しない |
| 6 | ドキュメントと実装のずれ | 混乱・手戻り | 実装を「正」としてドキュメントを随時修正する。同じ内容を複数ファイルに書かない（分冊・重複なし方針） |
